# WoClaw 维护规则

## CI/CD 优先级规则

### 核心原则：CI/CD 失败 > 所有其他事项

**当 CI/CD 构建失败时，必须立即中断所有待处理的开发工作，优先解决 CI/CD 问题。**

```
CI/CD 失败 → 暂停一切 → 定位根因 → 修复 → 验证通过 → 恢复开发
```

### 具体规则

1. **CI/CD 失败时禁止的操作**
   - ❌ 继续开发新功能
   - ❌ 合并其他 PR
   - ❌ 处理非紧急 issue
   - ❌ 启动新的子任务

2. **CI/CD 失败时的正确操作**
   - ✅ 查看失败日志，定位失败原因
   - ✅ 如果是自身提交的代码导致的失败，立即修复
   - ✅ 如果是依赖或环境问题，更新依赖或修复环境
   - ✅ 如果是基础设施问题（如 GitHub Actions 配置），优先修复
   - ✅ 验证修复后确保 CI/CD 全部通过

3. **恢复条件**
   - 所有 workflow 状态为 ✅ PASSING
   - 没有其他阻塞性问题待处理

### GitHub Actions 工作流

项目使用以下 CI/CD 工作流：

| Workflow | 触发条件 | 职责 |
|----------|----------|------|
| `ci.yml` | push/PR | 单元测试、类型检查、lint |
| `docker.yml` | push to main | 构建 Docker 镜像 |
| `publish.yml` | tag | 发布 npm 包 |

### 分支保护

- `main` 分支需要通过所有 CI 检查才能合并
- 禁止强制推送 `main` 分支

---

*此规则旨在确保代码库始终处于可发布状态，避免问题累积和回归。*
